# cursor-top Performance Baseline

Canonical "numbers to beat" for the `cursor-top UX, Features & Performance
Optimisation` spec (`specs/20260610-cursor-top-ux-perf/`). Recorded by
subtask 01 **before any production change** — the `src/` tree at measurement
time is the pre-optimisation pipeline (fresh collector per tick, full re-read
of every session file, windowed re-tail of every log source, full re-render).
Subtasks 03 (collector caching), 07 (render windowing), and 08 (integration)
append before/after sections below and must publish measured deltas against
these numbers.

## Machine context

| | |
|---|---|
| CPU | Intel(R) Core(TM) i7-9750H @ 2.60GHz (6 cores / 12 threads) |
| RAM | 30 GiB |
| OS | Ubuntu 24.04.4 LTS, kernel 6.17.0-35-generic (linux x86_64) |
| Node | v22.16.0 |
| pnpm | 10.8.0 |
| vitest | 4.1.6 (`vitest bench`, tinybench-based) |
| Date | 2026-06-10 (UTC) |

Numbers below are tinybench `mean` values in **milliseconds** (lower is
better) from a single committed run; `rme` was ≤ ~15 % on the slowest
benches (10-sample L-tier e2e) and ≤ ~3 % elsewhere. Run-to-run variance
of ±20–40 % was observed on this shared desktop between back-to-back
suite runs — treat sub-2× deltas with suspicion and re-measure
before/after pairs in the same sitting on an otherwise idle machine.

## Collector: end-to-end `collect()` latency per tick (mean ms)

Warm process, collector instance reused across iterations, injected
fs/psRunner/composerModelRunner (no real `ps`, no real `sqlite3`, no real
Cursor state dirs; transcript tails hit real temp files — see limitation).

| Tier | default flags | prunes off |
|---|---|---|
| S | 4.66 | 3.32 |
| M | 40.6 | 32.4 |
| L | **281** | 328 |

At tier L a 1 s refresh interval spends ~28 % of wall time inside
`collect()` — the headline number subtask 03 must reduce. (The two
variants bracket the prune-pass cost; differences between them at M/L are
within run noise because pruning itself is cheap — the dominant cost is
tailing, below.)

## Collector: stage-level timings (mean ms)

| Stage | S | M | L |
|---|---|---|---|
| `readSessionRecords` (cli roots) | 0.011 | 0.022 | 0.040 |
| `readTranscriptRecords` (walk + stat + mint) | 0.141 | 0.527 | 5.17 |
| tail fan-out (`tailJsonlMessages` + `tailFile`, all log sources) | 3.40 | 28.1 | **259** |
| `buildHierarchy` (incl. sort) | 0.011 | 0.090 | 0.786 |

The tail fan-out **is** the tick at every tier (~73–92 % of e2e): every
tick re-opens and re-reads the trailing window (64 KB for `.jsonl`
transcripts, 16 KB for plain logs) of every log source — ~1000 opens and
up to ~64 MB read per tick at tier L. This is the mtime-gating target for
subtask 03.

## Collector: fs operations per tick (default flags)

Counted by the `CountingFs` wrapper in `bench/fixtures.ts` over one
`collect()` with CLI-default flags (deterministic — the pipeline is
cache-free today, so every tick repeats these):

| Tier | readdir | readFile | stat | exists | total FsLike ops | ps spawns | sqlite3 spawns |
|---|---|---|---|---|---|---|---|
| S | 17 | 12 | 17 | 33 | 79 | 1 | 1 |
| M | 45 | 107 | 133 | 219 | 504 | 1 | 1 |
| L | 273 | 1016 | 1256 | 2034 | **4579** | 1 | 1 |

> **Known limitation (tail-seam):** `tailFile` / `tailJsonlMessages`
> (`src/discovery/logs.ts`) open log files via `node:fs/promises.open`
> directly, bypassing the injected `FsLike` — their `open()`/`stat()`/
> `read()` I/O (one of each per log source per tick: ~1001 extra opens +
> reads at tier L) is **not** included in the FsLike counts above.
> Subtask 03 introduces the observable tail seam and reports tail reads
> as a separate metric column.

## Renderer timings (mean ms)

Snapshots produced once per tier by a real `collect()` (default flags),
rendered repeatedly with a fixed clock.

| Tier | Visible rows | `renderText(snapshot)` | row-build (`flattenVisible` + `formatAgentRowLine` per row) |
|---|---|---|---|
| S | 9 | 0.023 | 0.014 |
| M | 86 | 0.204 | 0.142 |
| L | 822 | 3.27 | 1.28 |

Pure string-building stays in single-digit milliseconds even at 822 rows;
the interactive cost subtask 07 attacks is Ink/React re-rendering every
row every second, for which the row-build path above is the per-row lower
bound (windowing caps how many rows pay it).

## How to re-run

```sh
# from the repo root
pnpm --filter @zoto-agents/zoto-cursor-top run bench

# contract regression tests (must stay green for every spec subtask)
pnpm --filter @zoto-agents/zoto-cursor-top exec vitest run tests/contracts.test.ts
```

`vitest bench` only collects `bench/*.bench.ts` files; the normal test
suite (`pnpm --filter @zoto-agents/zoto-cursor-top test`) does not execute
benches. Fixture temp directories are created under `os.tmpdir()`
(`cursor-top-bench-<tier>-*`) and removed on process exit. The benches
never read real Cursor state directories and never spawn real processes.

Per-tier fixture composition, post-prune node counts, and the
`[fs-ops <tier>]` table are printed during bench setup — copy them from
the run output when updating this file.

## Fixture / harness API (for later subtasks)

`bench/fixtures.ts` exports:

- `createScaleFixture(tier)` → `ScaleFixture` with `fs` (a `CountingFs`
  wrapping the in-memory `MemFs`), `psRunner`, `composerModelRunner`,
  `pidParents`, `transcriptRoots`, `transcriptFiles`, `plainLogFiles`,
  `counts`, `calls` (ps/composer spawn counters), `modelFor(id)`,
  `collectorOptions(overrides?)` (CLI-default flags pre-wired), and
  `dispose()`.
- `CountingFs` — `FsLike` wrapper counting `readdir`/`readFile`/`stat`/
  `exists`; `reset()`, `total()`.
- `MemFs` — in-memory `FsLike` (`addDir`/`addFile`, metadata-only mirrors
  supported for real temp files).
- `registerExitCleanup(fixture)` — exit-hook temp-dir cleanup for bench
  files (tests should prefer `await fixture.dispose()` in `afterAll`).
- `TIERS`, `TIER_CONFIG`, `chatUuid`/`subUuid`, `mulberry32`.

### Post-subtask-07 (render windowing)

Measured 2026-06-11 (Node v22.16.0) via `pnpm exec tsx bench/quick-render-window.mjs`.
Tier **S** uses the real scale fixture collector; **M/L** use in-memory synthetic
snapshots with the baseline visible-row counts (86 / 822) and comfortable-density
row shape (row + title + 3 log lines) — same approach as the subtask-03 deferral
for slow cold collects. Timed loop covers **viewport slice formatting only**
(`formatAgentRowLine` for windowed rows); flatten + window math is precomputed
outside the loop (matches `App.tsx` where `visible` is `useMemo`'d).

Viewport: 40-line body budget (`terminalRows=48`, default chrome, 2-line
indicator reserve) → **8 agent rows** per window at comfortable density.

| Tier | Visible rows | Window rows | Full row-build (mean ms) | Windowed row-build (mean ms) | vs full | L/M windowed |
|---|---|---|---|---|---|---|
| S (fixture) | 9 | 9 | 0.040 | 0.062 | 1.5× (tree fits window) | — |
| M (synthetic) | 86 | 8 | 0.265 | **0.029** | **0.11×** | — |
| L (synthetic) | 822 | 8 | 2.158 | **0.016** | **0.007×** | **0.55× M** ✓ |

Windowed frame-build is **O(viewport)** — tier-L windowed time is **below**
tier-M (0.016 ms vs 0.029 ms), well under the 1.5× target. Both are far below
the unwindowed full-tree row-build at the same tier (M 0.265 ms → 0.029 ms;
baseline fixture full row-build was 0.142 ms / 1.28 ms at M/L). Ink/React
overhead is not included; the interactive win is capping Row mounts to ~8 nodes
plus `React.memo` with quantised `startColumn` props.

Quick re-run (no vitest bench loop):

```sh
pnpm exec tsx bench/quick-render-window.mjs        # S + M + L
pnpm exec tsx bench/quick-render-window.mjs M      # single tier
```

Full `pnpm run bench` render tier (includes `windowed row-build` benches for M/L
in `bench/render.bench.ts`) deferred to subtask 08 integration verification.
