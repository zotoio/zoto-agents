# Subtask: Engine `runner.ts` + `case.ts` + `update.ts` for Runner Cases

## Metadata
- **Subtask ID**: 04
- **Feature**: evals-json-first-migration
- **Assigned Subagent**: generalPurpose
- **Dependencies**: 01
- **Created**: 20260527

## Objective

Teach the declarative-side engine (`engine/runner.ts`, `engine/case.ts`, `engine/update.ts`) to recognise the new `runner` field on JSON cases. The declarative path remains responsible for `eval:list`, `eval:judge-only`, and the future `eval:update` drift detection — but it must NOT try to execute runner cases. Vitest owns that.

## Deliverables Checklist
- [x] `engine/case.ts`:
    - Extend `EvalCase` with `runner?: string` and `parameters?: Record<string, unknown>` (already established in subtask 01; this subtask CONSUMES that field).
    - Strengthen `validateEnriched` to:
        - Treat a runner case as valid even when `prompt`, `assertions`, `graders`, `fixtures`, `expected_filesystem` are absent.
        - Reject hybrid cases (runner + any declarative field).
        - Reject empty `runner` or non-`.test.ts` runner extensions.
    - Expose a small helper `isRunnerCase(c: EvalCase): boolean` re-usable by the runner and updater.
- [x] `engine/runner.ts`:
    - In `discoverCaseFiles()` / the main case-loading path, after loading every JSON file, partition cases into `declarative[]` and `runner[]`.
    - For runner cases:
        - When invoked with `--list`: print them as `<targetId>#<caseId> [runner: <path>]` so the operator sees coverage.
        - When invoked with `--full` (declarative run): SKIP runner cases entirely. Emit one summary log line per target: `[runner-cases] target=<id> count=<n> deferred-to-vitest`.
        - The `llm.yml` written by the declarative path MUST NOT contain rows for runner cases (avoids double-reporting; the Vitest reporter writes those rows).
    - When invoked with `--judge-only`: leave runner rows in `llm.yml` (if any are present from a Vitest run) untouched — judge-only is read-only.
- [x] `engine/update.ts`:
    - Drift detection now compares JSON file checksums. Runner cases are user-authored by definition (a generated declarative case becoming a runner case would require explicit user action) — preserve them verbatim during regeneration. Confirm `_meta.generated: true` runner cases are not regenerated by the analyser (the analyser only emits declarative cases).
    - The `regenerateLlm` codepath still writes declarative cases, but its output is now `<name>.json` (subtask 08 finalises the writer; this subtask just ensures the data model supports it).
    - Layout-drift warnings: if `update.ts` discovers a co-located `.test.ts` LLM eval that is NOT a scenario file (i.e. lives under `<kind>/evals/*.test.ts`), treat it as **critical drift** (exit 2 in `--check` mode) starting in subtask 10's CI; this subtask just adds the detection helper `findCoLocatedTsEvals()` and writes a deprecation-warning log line for any matches.
    - Preserve `runner` cases when surgical-merging analyser output into existing JSON: walk the existing cases, retain any with `_meta.generated !== true` (which includes runner cases), and append/update generated ones.
- [x] `engine/_user-case-guards.ts`: confirm `isUserAuthoredCase` already returns `true` for any case without `_meta.generated === true`. No change expected, but add a focused test that a runner case with `_meta.generated: false` is treated as user-authored.
- [x] Unit tests:
    - `engine/runner.ts`: `--list` includes runner cases; `--full` skips them; deferred-log line is emitted with correct count.
    - `engine/case.ts`: `isRunnerCase` and the hybrid-rejection branch.
    - `engine/update.ts`: drift detection treats `.test.ts` co-located evals as deprecation warnings; runner cases survive surgical merge.

## Implementation Notes

- **Why preserve runner cases as user-authored:** Runner cases reference external TS files written by humans. Regenerating them from the analyser would be wrong — the analyser only knows declarative shapes. The `_meta.generated !== true` rule already covers this, so the only work here is making sure the surgical-merge code path does not accidentally clobber them.
- **`--list` output format:** Existing format is one line per case. Keep it parseable. Adding a trailing `[runner: <path>]` token is non-breaking for current consumers. Update `scripts/eval-list.ts` only if it parses the output (likely it just reprints it).
- **No runtime execution in `engine/runner.ts` for runner cases:** Confirm that no code path tries to invoke the SDK for a runner case. The simplest implementation filters them out at the top of the case-loop after partition.
- **`findCoLocatedTsEvals()` helper:** Glob `**/{commands,agents,hooks}/evals/*.test.ts` under repo roots configured in the manifest (`plugins/*`, `.cursor/`). Exclude `evals/scenarios/**` and `evals/llm/_shared/**`. Returns absolute paths.
- **Deprecation warning text (this subtask):** `"[layout-drift] Co-located TS LLM eval found at <abs path>; this file should be migrated to JSON (see spec evals-json-first-migration-20260527)."` — does NOT exit non-zero yet.
- **Schema consumption:** Ajv-validate every JSON file against `templates/schema/eval-file.schema.json` (created in subtask 01) on load. Reject files that fail validation with a path-prefixed error.
