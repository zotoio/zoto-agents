zoto-agents

Plan and verify your specs. Generate and update your evals.

Two Cursor plugins, shipped together as peers. zoto-spec-system decomposes complex initiatives into reviewable specs and runs them with adversarial verification. zoto-eval-system stamps static and LLM eval suites side-by-side and updates them with your confirmation when AI primitives change.

How It Works

A short, structured loop per plugin. Run them independently; pair them when a feature has both spec and eval drift.

Spec System — create → judge → execute

1
Create

Run /z-spec-create to decompose a feature into phased subtasks with dependencies, deliverables, and definitions of done.

2
Judge

Run /z-spec-judge for an independent assessment. The judge reviews feasibility, risks, and completeness before execution.

3
Execute

Run /z-spec-execute to implement subtasks in order. Each deliverable is adversarially verified before moving on.

Eval System — create → execute → update / judge / advise / compare

1
Create

Run /z-eval-create to scaffold pytest and @cursor/sdk backends, generate cases, and stamp the manifest.

2
Execute

Run /z-eval-execute to capture a timestamped run with static, LLM, and aggregate report.yml outputs.

3
Update / Judge / Advise / Compare

Detect drift, review and approve eval updates per-change, surface weak graders, recommend the next move, and chart cross-run results on a /canvas.

Commit Specs and Eval Cases Alongside Your Code

Specs land as durable markdown in your repo so the why behind a change survives the chat session. Eval cases land as runnable pytest and @cursor/sdk suites so the how-do-we-know-it-works moves with the code. Pair either with CRUX Memories to compound learnings across sessions.

Install

Each plugin installs independently. Add one, both, or neither — they cooperate but never gate each other.

cursor plugin install zoto-spec-system
cursor plugin install zoto-eval-system
